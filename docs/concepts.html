🧠 my_agents — концепции для чайников

Кто что делает и чем агент отличается от скилла, команды, плагина и MCP-инструмента.

⚡ TL;DR за 30 секунд

my_agents — «коробка с готовой IT-командой» для Claude Code. Главное: агент/скилл/команда — это текст-инструкции (кем быть / как делать / ярлык); MCP-инструмент — это код-действие (реально сделать); плагин — упаковка, которая всё это привозит.

🏢 Аналогия: IT-компания

  • 🤖 Агент — нанятый эксперт с должностной инструкцией. Ставишь задачу — он думает и делает. Пример: code-review-swarm.
  • 🧩 Скилл — корпоративный плейбук «как провести код-ревью». Любой сотрудник открывает и следует. Пример: github-code-review.
  • ⌨️ Команда — кнопка на пульте: нажал /init — запустился сценарий. Пример: /security-review.
  • 🔌 Плагин — коробка «отдел безопасности под ключ»: привозит своих сотрудников (агентов), плейбуки (скиллы) и кнопки (команды). Пример: ruflo-security-audit.
  • 🛠️ MCP-инструмент — рабочий станок: «сохранить в БД», «создать Pull Request». Не инструкция, а само действие. Пример: memory_store, swarm_init.

👔 Кто кем командует (оргструктура агентов)

Не у каждого агента есть «руководитель» — руководитель это роль, а не обязательный атрибут. Из 121 агента ~28 — координаторы (🎖), остальные ~93 — специалисты-исполнители.

Tier 0 — боссы        queen-coordinator · collective-intelligence-coordinator ·
                      project-coordinator · v3-queen-coordinator · dual-orchestrator
   │  владеют целью, собирают команду
Tier 1 — топологии    hierarchical · mesh · adaptive-coordinator · sparc-coord
   │  выбирают форму команды, раздают работу
Tier 2 — консенсус    raft · byzantine · quorum · gossip · crdt
   │  согласуют общее состояние
Tier 3 — узкие спецы  load-balancer · performance-monitor · swarm-memory-manager
   │
Исполнители (без своего лида)   coder · reviewer · tester · backend-dev · game-dev-студия …

Почему у части направлений «—» в колонке «Руководители»?

Колонка показывает координаторов, которые физически лежат в этой папке-направлении. В core (coder/planner/researcher/reviewer/tester) координаторов нет — это все исполнители, поэтому «—».

Но они НЕ без руководства! Ими командуют боссы из других направлений (project-coordinator, queen-coordinator, топология-координаторы). Координация сквозная, а не внутри каждой папки. Пустое «—» = «в этой папке нет начальника; это доводчики, которыми рулят верхние координаторы».

Честная оговорка: метка 🎖 ставится эвристикой по имени/направлению — например performance-benchmarker помечен «лидом», хотя по сути это Tier-3 измеритель. Настоящие «боссы» — координаторы/оркестраторы/queen.

🖥️ Что за панель внизу терминала (и зачем она)

Когда работаешь в Claude Code, внизу видна статус-панель AlexKo — «приборная доска» проекта. Её рисует маленький скрипт .claude/helpers/statusline.cjs, и она показывает живое состояние (обновляется само), а не статичную картинку.

Зачем нужна: с одного взгляда видно, что происходит — не «летишь вслепую». Какая модель, сколько занято контекста, идёт ли команда агентов, сколько хуков/тестов, есть ли уязвимости.

Что значат строки (сверху вниз):

  • Шапка — ▊ AlexKo версия · твоё git-имя · модель (Opus/Sonnet) · ⏱ время сессии · ● N% ctx — насколько заполнено «окно памяти» разговора (зелёный→жёлтый→красный; на 90% пора закругляться).
  • 🏗️ DDD Domains — прогресс 5 доменов проекта (0/5) + индикатор поиска по памяти.
  • 🤖 Swarm — активные агенты [0/15], хуки 🪝, уязвимости CVE, размер памяти 💾, % обученности 🧠.
  • 🔧 Architecture — ADR-решения, прогресс DDD, статус безопасности.
  • 📊 AgentDB — сколько векторов в памяти, размер БД, сколько тест-файлов, интеграции (MCP/БД).

💲 Стоимость скрыта по умолчанию (это приблизительная клиентская оценка — на подписке вводит в заблуждение). Включить обратно: переменная RUFLO_STATUSLINE_SHOW_COST=1. Нижняя строка ⏵⏵ accept edits — это режим самого Claude Code, не часть панели. Полная расшифровка — в README.

🔄 Как это всё работает вместе

  1. Ты пишешь задачу. (опц.) срабатывает хук маршрутизации.
  2. Claude (ведущий) на сложной задаче зовёт координатора (🎖) или сразу спавнит нужных агентов.
  3. Агенты следуют скиллам (рецептам) и при необходимости дёргают MCP-инструменты (сохранить в память, создать PR, поиск).
  4. Память хранит общий контекст; агенты передают работу по конвейеру researcher → architect → coder → tester → reviewer.
  5. Плагины привозят больше агентов/скиллов/команд, если их установить.
  6. Команды (/...) — твой ручной быстрый запуск любого сценария.
