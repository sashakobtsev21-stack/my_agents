# 🧠 my_agents — концепции для чайников

Полное объяснение «кто что делает» и чем **агент** отличается от **скилла**, **команды**, **плагина** и **MCP-инструмента**.

---

## ⚡ TL;DR за 30 секунд

my_agents — это «коробка с готовой IT-командой» для Claude Code. Внутри 5 типов деталей:

| Деталь | Это… | Одной фразой |
|---|---|---|
| 🤖 **Агент** | сотрудник-специалист | **кто** делает (роль: «бэкендер», «ревьюер») |
| 🧩 **Скилл** | плейбук-рецепт | **как** делать процедуру (пошагово) |
| ⌨️ **Команда** | горячая клавиша `/...` | **быстрый запуск** готового сценария |
| 🔌 **Плагин** | готовый «отдел» | **бандл** агентов+скиллов+команд по теме |
| 🛠️ **MCP-инструмент** | станок / API | **реальное действие** (сохранить в память, создать PR) |

**Ключевое различие:** агент/скилл/команда — это **текст-инструкции** (кем быть / как делать / ярлык). MCP-инструмент — это **код-действие** (реально сделать). Плагин — **упаковка**, которая всё это привозит.

---

## 🏢 Аналогия: IT-компания

- 🤖 **Агент** — нанятый эксперт с должностной инструкцией. Ставишь задачу — он думает и делает. *Пример: `code-review-swarm` — старший ревьюер.*
- 🧩 **Скилл** — корпоративный плейбук «как провести код-ревью». Любой сотрудник открывает и следует. *Пример: `github-code-review`.*
- ⌨️ **Команда** — кнопка на пульте: нажал `/init` — запустился сценарий. *Пример: `/security-review`.*
- 🔌 **Плагин** — коробка «отдел безопасности под ключ»: привозит своих сотрудников (агентов), плейбуки (скиллы) и кнопки (команды). *Пример: `ruflo-security-audit`.*
- 🛠️ **MCP-инструмент** — рабочий станок: «сохранить в БД», «создать Pull Request», «искать по памяти». Не инструкция, а само действие. *Пример: `memory_store`, `swarm_init`.*

## 👔 Кто кем командует (оргструктура агентов)

**Не у каждого агента есть «руководитель» — потому что руководитель это роль, а не обязательный атрибут.** Из 104 агентов часть — координаторы (🎖, тиры 0–1), большинство — специалисты-исполнители (тиры 2–3).

```
Tier 0 — боссы          queen-coordinator · collective-intelligence-coordinator ·
                        project-coordinator · v3-queen-coordinator · dual-orchestrator
   │  владеют целью, собирают команду
Tier 1 — топологии      hierarchical · mesh · adaptive-coordinator · sparc-coord
   │  выбирают форму команды, раздают работу
Tier 2 — консенсус      raft · byzantine · quorum · gossip · crdt
   │  согласуют общее состояние
Tier 3 — узкие спецы    load-balancer · performance-monitor · swarm-memory-manager
   │
Исполнители (без своего лида)   coder · reviewer · tester · backend-dev · game-dev-студия …
```

### Почему в таблице у части направлений «—» в колонке «Руководители»?

Колонка показывает координаторов, которые **физически лежат в этой папке-направлении**. В направлении `core` (`coder`, `planner`, `researcher`, `reviewer`, `tester`) координаторов нет — это все исполнители, поэтому «—».

**Но они НЕ без руководства!** Ими командуют боссы из *других* направлений (`project-coordinator`, `queen-coordinator`, топология-координаторы). Координация **сквозная** — а не внутри каждой папки. Пустое «—» значит «в этой папке нет начальника; это доводчики, которыми рулят верхние координаторы».

**Направления с координаторами (🎖):** consensus, hive-mind, swarm, github, templates, dual-mode, sublinear, v3, game-dev (`game-director`), root (`project-coordinator`).
**Только исполнители (—):** core, flow-nexus, optimization, sparc, development, devops, data, testing, analysis, goal, и одиночные (neural, sona, reasoning, documentation, architecture, payments, custom, specialized).

> Честная оговорка: метка 🎖 ставится эвристикой по имени/направлению — например `performance-benchmarker` помечен как «лид», хотя по сути это Tier-3 измеритель. Настоящие «боссы» — это координаторы/оркестраторы/queen.

## 🔄 Как это всё работает вместе (один поток)

1. Ты пишешь задачу. *(Опц.)* срабатывает **хук** маршрутизации.
2. Claude (ведущий) на сложной задаче зовёт **координатора** (🎖 агент) или сразу спавнит нужных **агентов**.
3. Агенты следуют **скиллам** (рецептам) и при необходимости дёргают **MCP-инструменты** (сохранить в память, создать PR, поиск).
4. **Память** хранит общий контекст; агенты передают работу друг другу по конвейеру (`researcher → architect → coder → tester → reviewer`).
5. **Плагины** просто привозят больше агентов/скиллов/команд, если их установить.
6. **Команды** (`/...`) — твой ручной быстрый запуск любого сценария.
